# English Words Constitution

## Core Principles

### I. テスト駆動開発（TDD）（NON-NEGOTIABLE）

テスト駆動開発は本プロジェクトの必須開発手法である。例外は認めない。

- テストは実装コードよりも先に記述しなければならない（MUST）。
- テストが失敗（Red）することを確認してから、実装コードを書く。
- Red-Green-Refactor サイクルを厳守する:
  1. 期待する振る舞いを定義する失敗テストを書く。
  2. テストを通す最小限のコードを書く。
  3. テストが全て通る状態を保ちながらリファクタリングする。
- すべてのユーザー向け機能および内部モジュールは、完了とみなす前に
  対応するテストカバレッジを持たなければならない（MUST）。
- フロントエンド: React Testing Library / Jest を使用する。
- バックエンド: NestJS の組み込みテスト機能（Jest ベース）を使用する。

**根拠**: TDD は設計品質の高い、回帰に強いコードを生み出す。
英単語学習アプリでは正確な動作がユーザーの学習体験に直結するため、
テスト先行は不可欠である。

### II. RESTful API 設計

バックエンド API は RESTful 設計を基本とする。

- リソース指向の URL 設計を採用する（例: `/api/words`, `/api/words/:id`）。
- HTTP メソッドを正しく使い分ける（GET: 取得、POST: 作成、
  PUT/PATCH: 更新、DELETE: 削除）。
- レスポンスには適切な HTTP ステータスコードを返す（MUST）。
- エラーレスポンスは統一されたフォーマットで返す（MUST）。
- API のバージョニングが必要になった場合は URL プレフィックスで
  管理する（例: `/api/v1/`）。

**根拠**: RESTful 設計はフロントエンド（React）とバックエンド
（NestJS）の間に明確な契約を提供し、独立した開発・テストを
可能にする。

### III. データ整合性

単語データ、学習状況、ユーザー情報は常に正確かつ整合性を
保たなければならない。

- すべての外部入力（ユーザー入力、API リクエスト）はシステム境界で
  バリデーションしなければならない（MUST）。
- 学習状況の遷移（未学習 → 学習中 → 習得済み）は明確に定義し、
  不正な状態遷移を許容してはならない（MUST NOT）。
- 単語データ（スペル、意味、例文）は登録・更新・取得の過程で
  情報を欠落させてはならない（MUST NOT）。
- データベーススキーマの変更にはマイグレーションを必ず含める。
  データ損失は一切認めない。
- MySQL の外部キー制約・ユニーク制約を活用し、DB レベルでも
  整合性を担保する（MUST）。

**根拠**: 英単語学習ツールはコンテンツの正確さと進捗管理の
信頼性が価値の根幹である。データが不正確であればユーザーの
信頼を失う。

## 品質基準

- **言語**: TypeScript（フロントエンド・バックエンド共通）
- **フロントエンド**: React
- **バックエンド**: NestJS
- **データベース**: MySQL
- **インフラ**: Docker
- **テスト**: Jest（フロントエンド・バックエンド共通）
- **ドキュメント**: 対話・ドキュメントはすべて日本語で記述する（MUST）
- **コミット規律**: 各コミットは単一の論理的変更を表す（SHOULD）。
  main ブランチに失敗するテストを含むコミットを入れてはならない
  （MUST NOT）。

## 開発ワークフロー

1. **main からブランチを作成** して機能開発・修正を行う。
2. **テストを先に書く**（原則 I に従う）。
3. **最小限のコードを実装** してテストを通す。
4. テストが全て通る状態を保ちながら **リファクタリング** する。
5. **レビュー**: すべての PR は constitution 準拠を確認してから
   マージする（MUST）。
6. **main にマージ**: 全テストが通り、レビューが承認された場合のみ。

## Governance

- 本 constitution は、競合が生じた場合、他のすべての開発慣行・
  規約に優先する。
- 改定には以下を要する:
  1. 変更内容と根拠を記述した提案書。
  2. セマンティックバージョニングに従ったバージョン更新:
     - MAJOR: 原則の削除または互換性のない再定義。
     - MINOR: 新原則の追加または既存ガイダンスの実質的拡張。
     - PATCH: 文言の修正、誤字脱字、意味を変えない調整。
  3. 本ファイルの更新と依存テンプレートへの伝播。
- すべての PR およびコードレビューは本 constitution の原則への
  準拠を検証しなければならない（MUST）。
- `.specify/memory/constitution.md` がプロジェクトガバナンスの
  唯一の真実の源（single source of truth）である。
